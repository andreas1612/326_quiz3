# Nektarios Binaries: Exploit Report & Methodology

This document summarizes the methodology and exact parameters used to successfully exploit binaries `bin.1`, `bin.2`, and `bin.3` in the `nektarios` folder. Full shell execution (`uid=9992`) was achieved on the lab machine on the first attempt.

> **Note:** `bin.0` is a PIE (`ET_DYN`) warm-up binary with no `display_file`/`memcpy` vulnerability — skipped per quiz instructions (bin.1–bin.3 only).
> **Note:** The `nektarios` binaries are **identical** to the `lefteris` set (same BuildIDs). All addresses and offsets match exactly.

---

## 1. The Core Environment Rule

All binaries were run under:
```bash
env -i TEMP=1000 setarch i686 -R --3gb ./binary [payload]
```
This disables ASLR and seeds `srandom()` via `TEMP=1000` for a deterministic stack layout.

---

## 2. Methodology & Reconnaissance

### Binary Classification
Using `readelf -l ./binary | grep GNU_STACK` and `readelf -h ./binary | grep Type`:

| Binary | Stack | Type | Attack |
|--------|-------|------|--------|
| bin.0 | RW | ET_DYN (PIE) | Warm-up — skip |
| bin.1 | RW (NX on) | ET_EXEC | Ret2libc |
| bin.2 | RWE (NX off) | ET_EXEC | Shellcode injection |
| bin.3 | RWE (NX off) | ET_EXEC | Shellcode injection |

No stack canaries (`__stack_chk_fail`) in any binary.

### Vulnerability
All exploitable binaries use the same pattern in `display_file()`:
- `fscanf` reads a size value into a stack variable
- `malloc(size)` allocates a heap buffer
- `fgetc` loop fills heap buffer from file
- **`memcpy(stack_buf, heap_buf, size)`** — no bounds check → stack buffer overflow

### Offset Calculation
From `objdump -d` of each binary's `display_file`:

```asm
# bin.1:  lea -0x2a(%ebp), %eax  → buffer at ebp-42 → offset = 42+4 = 46
# bin.2:  lea -0x2c(%ebp), %eax  → buffer at ebp-44 → offset = 44+4 = 48
# bin.3:  lea -0x28(%ebp), %eax  → buffer at ebp-40 → offset = 40+4 = 44
```

Address after `memcpy` call (used as GDB breakpoint): `0x804933c` (all three binaries).

## 6. Notes

- Both `lefteris` and `nektarios` sets contain **identical binaries** (same BuildIDs: bin.1=`9d5944c7`, bin.2=`c66c02a8`, bin.3=`f394a776`). Exploit files are the same for both sets.
- Each binary in the set has a **different buffer offset** (`-0x2a`, `-0x2c`, `-0x28`) — not all the same.
- No `mmap`/gadget-building in any `main()` — pure ret2libc / shellcode, no ROP needed.
- bin.2 and bin.3 are both RWE but have different offsets and thus different buf_addrs.
