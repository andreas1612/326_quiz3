# Kyriaki Binaries: Exploit Report & Methodology

This document summarizes the methodology and exact parameters used to successfully exploit binaries `bin.1`, `bin.2`, and `bin.3` in the `kyriaki` folder. Full shell execution (`uid=9992`) was achieved on the lab machine on the first attempt.

> **Note:** `bin.0` is a PIE (`ET_DYN`) warm-up binary — skipped per quiz instructions (bin.1–bin.3 only).

---

## 1. The Core Environment Rule

All binaries were run under:
```bash
env -i TEMP=1000 setarch i686 -R --3gb ./binary [payload]
```
This disables ASLR and seeds `srandom()` via `TEMP=1000` for a deterministic stack layout.

---

## 2. Methodology & Reconnaissance

### Binary Classification
Using `readelf -l ./binary | grep GNU_STACK` and `readelf -h ./binary | grep Type`:

| Binary | Stack | Type | Attack |
|--------|-------|------|--------|
| bin.0 | RW | ET_DYN (PIE) | Warm-up — skip |
| bin.1 | RW (NX on) | ET_EXEC | Ret2libc |
| bin.2 | RWE (NX off) | ET_EXEC | Shellcode injection |
| bin.3 | RWE (NX off) | ET_EXEC | Shellcode injection |

No stack canaries (`__stack_chk_fail`) in any binary. No `mmap`/gadget-building in any `main()`.

### Vulnerability
All exploitable binaries use the same pattern in `display_file()`:
- `fscanf` reads a size value into a stack variable
- `malloc(size)` allocates a heap buffer
- `fgetc` loop fills heap buffer from file
- **`memcpy(stack_buf, heap_buf, size)`** — no bounds check → stack buffer overflow

### Offset Calculation
From `objdump -d` of each binary's `display_file` (breakpoint at `0x804933c` — right after `call memcpy@plt`):

```asm
# bin.1:  lea -0x2e(%ebp), %eax  → buffer at ebp-46 → offset = 46+4 = 50
# bin.2:  lea -0x30(%ebp), %eax  → buffer at ebp-48 → offset = 48+4 = 52
# bin.3:  lea -0x2c(%ebp), %eax  → buffer at ebp-44 → offset = 44+4 = 48
```

## 6. Notes

- This set shares the same vulnerability structure as `lefteris` and `nektarios` (display_file + memcpy overflow).
- Each binary has a **different buffer offset** (`-0x2e`, `-0x30`, `-0x2c`) — not all the same. Always check each binary individually.
- Both bin.2 and bin.3 are RWE (2 shellcode bins out of 3) — different from regina/regina2/tasos sets which had only 1 RWE binary.
- Libc addresses match the lefteris/nektarios set (`system=0xb7dffd30`, `/bin/sh=0xb7f40caa`).
- buf_addrs differ from lefteris/nektarios despite identical offsets — always obtain from lab machine via GDB.
